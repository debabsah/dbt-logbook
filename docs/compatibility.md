# dbt version compatibility

dbt-logbook reads only dbt's artifact files and shells out to the dbt CLI
where needed. It never imports dbt internals and never needs its own dbt
installation - which is why one version of dbt-logbook works across dbt Core
major versions.

## Tested versions

The ingest suite runs against golden artifacts generated by real dbt builds:

| dbt Core | manifest schema | run_results schema |
|---|---|---|
| 1.7 | v11 | v5 |
| 1.8 | v12 | v6 |
| 1.10 | v12 | v6 |
| 1.11 (current stable) | v12 | v6 |
| 2.0 (alpha, Rust engine) | v12 | v6 |

The fixtures live in `tests/fixtures/` and act as the regression net for
every future dbt release: when a new version ships, its artifacts get added
and the suite must stay green.

## dbt Core v2 notes

- v2 still writes `manifest.json` and `run_results.json` as JSON in the
  target dir, with the same schema versions as 1.11. The additional parquet
  artifacts in `target/data/` are not required by dbt-logbook.
- v2 removes some node fields (including `created_at`) and some result fields
  (including `failures`). Extraction is tolerant by design: absent fields
  become NULL, never errors.

## Known caveat: checksums across engine majors

The same model file hashes to different per-node checksums under dbt 1.x and
2.x engines. A diff whose two runs straddle an engine upgrade will therefore
report everything as modified once; the UI shows an engine-change banner on
that comparison instead of pretending the numbers are comparable.

## Unknown future versions

Ingest extracts a narrow set of shallow fields defensively and keeps the raw
gzipped artifact in the store. If a future dbt version renames fields, the
worst case is missing values in the UI until dbt-logbook catches up - and the
raw artifacts already stored can be re-extracted retroactively after an
upgrade, so no history is lost.
